## Setup workflow per new repo

When you start working in a new (40th) repo:

1. **Drop the bundle** — copy this `.github/` folder to the repo root (or use a sync script).
2. **Open the repo in VS Code or IntelliJ** with Copilot enabled.
3. **In Copilot Chat, type `/bootstrap-repo`** — Copilot will:
   - Read `package.json`, configs, sample tests, CI files, sample components
   - Generate `.github/instructions/repo-specific.instructions.md` tailored to this repo
   - Report findings + ask you any questions it couldn't answer from inspection
4. **Answer the open questions** Copilot asks; the file gets filled in.
5. **Commit `repo-specific.instructions.md`** — now this repo's Copilot is permanently aware of its specifics.

For subsequent sessions in this repo, just open and work — Copilot reads the full set automatically.

When the stack changes (dep upgrade, new pattern adopted), run `/refresh-repo-instructions` and Copilot updates the file.

## Recommended workflow per task type

### Starting a new feature
1. (One-time per repo) `/bootstrap-repo` if not done yet
2. Open Copilot Chat with `@workspace` then describe the feature
3. Drag-drop 1–2 sibling files for additional context
4. Ask Copilot to plan first, code second

### Writing an E2E test
1. `/write-e2e-test`
2. Drag-drop the component file(s) under test
3. Drag-drop one similar existing E2E test
4. Answer Copilot's context questions before any code is written

### Ready to commit
1. Stage your changes
2. `/pre-commit-check`
3. Address ❌ items, decide on ⚠️ items, commit using the suggested message

### Auditing existing code
1. Open the file(s)
2. `/review-existing-code`
3. Use the report to plan a follow-up — Copilot won't change anything during the audit

## Tips for the 40-repo problem

- **Generic files identical everywhere.** Use a script or internal template to sync this `.github/` folder. Differences belong only in `repo-specific.instructions.md`.
- **`repo-specific.instructions.md` is a living doc.** Re-run `/refresh-repo-instructions` after major changes. Treat it like a contributor onboarding doc.
- **One mental model across 40 repos.** Generic rules don't change → your habits don't change → Copilot's behavior is predictable.

## When Copilot still goes off the rails

- **Inventing imports / packages**: drag `package.json` into chat, OR run `/refresh-repo-instructions`.
- **Ignoring conventions**: drag 2–3 sibling files. If this happens often, the convention isn't captured — update `repo-specific.instructions.md`.
- **Writing E2E that won't pass**: use `/write-e2e-test`, never free-form for E2E.
- **Hallucinating DS components**: drag the DS package's `index.ts` into chat. Update `repo-specific.instructions.md` with the canonical exports.
- **Silently fixing things you didn't ask about**: remind it of section 4 of `copilot-instructions.md`.
- **Adding noise comments on obvious code**: point at `code-craft.instructions.md` "Comments" section.

## Plik 1: `.github/copilot-instructions.md`

````markdown
# Copilot Instructions — Banking Frontend Monorepo

> Read this file before EVERY response. These rules override your defaults.

## 1. Project Context

- **Domain**: Regulated banking application (compliance, PII protection, auditability matter)
- **Stack**: React 18+ · TypeScript (strict) · Vite · Vitest · Playwright · ESLint · Prettier
- **Structure**: Monorepo containing ~40 packages, including an internal `design-system` package
- **My environment**: VS Code in browser (cloud workspace) — **Chromium is NOT installed locally**, E2E tests run only in CI

## 2. Hard Rules (non-negotiable)

- TypeScript strict mode. No `any`. No `@ts-ignore` / `@ts-expect-error` without a justification comment on the line above.
- No `console.log`, `console.warn`, `console.error`, `debugger` in committed code. Use the existing logger.
- No new dependencies without checking `package.json` first. Always prefer existing libraries.
- No `localStorage` / `sessionStorage` for tokens, PII, or auth state.
- No hardcoded URLs, secrets, or environment values. Use `import.meta.env.VITE_*`.
- All new exported code must have unit tests.

## 3. Code Discovery Protocol — MANDATORY before generating code

Before producing ANY non-trivial code, do this in order:

1. **Read 2–3 sibling files** in the same folder to learn local conventions (imports, patterns, naming).
2. **Check the design-system package first** (`packages/design-system/` or similar in this monorepo) — never reinvent a Button, Modal, Input, Form, Table, or other UI primitive that already exists there. If unsure whether a component exists, ASK before creating one.
3. **Check barrel exports** (`index.ts` files) for existing utilities, hooks, and types.
4. **Identify existing patterns** for: state management, API/data fetching, error handling, routing, forms, validation.
5. **If your suggestion deviates from existing patterns**, you MUST flag it explicitly:
   > ⚠️ Deviation from existing pattern: [what], because [why]. Existing pattern is [X]. Confirm before I proceed.

## 4. Behavior Toward Existing Code

When I show you existing code, you MUST do BOTH:

**(a) Identify good patterns and preserve them.** Match indentation, import order, naming conventions, comment style, error handling style. Do not "modernize" or refactor unrelated code unless I asked.

**(b) Flag bad patterns you notice — do NOT silently fix them.** Surface these as a separate "Observations" section at the end:
- Direct DOM manipulation in React components
- `useEffect` with missing/wrong dependencies, or effects that should be derived state
- Untyped or `any`-typed API responses
- Tests asserting on implementation details (state, internal hook values) instead of user-visible behavior
- Missing error boundaries around async UI
- Inline styles or magic colors instead of design-system tokens
- Components > 200 lines or > 4 levels of nested JSX
- Logic that belongs in a custom hook, in a component
- Repeated literals that should be constants/enums
- Unsafe type casts (`as`) or non-null assertions (`!`) without justification

For each issue: state location, why it's a problem, and the minimal fix. Do not change code I didn't ask you to change.

## 5. Cross-Repo Awareness (Monorepo)

This monorepo has ~40 packages. When I open one package:

- **Do not assume types or utilities exist in other packages** unless I show you the import or you can verify in the visible files.
- **Do not invent package names** like `@bank/utils`, `@bank/api` — only reference packages you can actually see in `package.json` or import statements I've shared.
- **If you need something from another package and it's not visible**, ask me to share the relevant file rather than guessing.
- **Design-system components**: assume they exist for common primitives (Button, Input, Select, Modal, Form, Table, Toast, Alert, Card, etc.). Ask for the exact API if you need to use one you haven't seen.

## 6. Pre-Commit / Pipeline Verification

When I ask "ready to commit?", "review my changes", "check before push", or similar — run this checklist on the diff and report results in the format below.

### Checks to run mentally on the diff

1. **ESLint** — scan for: `react-hooks/exhaustive-deps`, `@typescript-eslint/no-floating-promises`, `@typescript-eslint/no-explicit-any`, unused imports/vars, missing return types on exported functions
2. **Prettier** — line length, trailing commas, quote style, semicolons (match `.prettierrc`)
3. **TypeScript** — trace types through changes; flag implicit `any`, unsafe casts, missing generic constraints
4. **Tests** — every changed source file must have corresponding test changes OR an explicit reason why not. New exported functions/components without tests = blocker.
5. **Build (Vite)** — flag new dynamic imports, env vars not in `.env.example`, circular dependencies
6. **E2E impact** — flag changes to: routes, ARIA roles/labels, `data-testid` attributes, form field names, button text used in selectors. These are likely to break Playwright tests.
7. **Bundle size risk** — flag new heavy imports (whole-library imports of lodash, moment, etc.) — suggest tree-shakeable alternatives
8. **Banking compliance**:
   - PII in logs (look for `logger.*` or `console.*` with user/account/card objects)
   - Hardcoded URLs to non-bank domains
   - Tokens/secrets in client storage or code
   - Missing input validation on user-supplied data
9. **SonarQube risks** — cognitive complexity > 15, duplicated blocks > 3 lines, deeply nested conditionals

### Required output format

```
## Pre-flight Check

✅ Will pass:
- [check name]: [why it's safe]

⚠️ Risks (likely warnings, won't block):
- [check name] at [file:line]: [issue] — suggested fix: [fix]

❌ Will fail:
- [check name] at [file:line]: [exact error CI will produce] — required fix: [fix]

📋 Manual verification needed:
- [thing I cannot verify from the diff alone]
```

If the diff has zero issues, say so — don't fabricate problems.

## 7. Asking for Context Instead of Guessing

When you don't have enough context, **ask** instead of inventing. Specifically, ask before:
- Choosing a state management approach (you don't know what we use)
- Picking an HTTP client (you don't know what we use)
- Naming components or files (you don't know our conventions)
- Adding new files outside the folder I'm working in
- Importing from a package you haven't seen me reference

Phrase questions concretely: "What's the existing pattern for X in this repo? Can you share `Y.tsx`?"

## 8. Companion Instruction Files

These files extend these rules — read whichever is relevant to the current task:

- `instructions/code-craft.instructions.md` — comments, naming, simplicity, error handling, security, money/dates, Definition of Done
- `instructions/react.instructions.md` — React/TS/Vite specifics
- `instructions/unit-tests.instructions.md` — Vitest + RTL
- `instructions/e2e.instructions.md` — Playwright (CRITICAL: no local Chromium)
- `instructions/design-system.instructions.md` — internal DS rules
- `instructions/pre-commit.instructions.md` — pipeline pre-flight format
- `instructions/repo-specific.instructions.md` — auto-generated per-repo overrides (ALWAYS read this if it exists; it captures THIS repo's actual stack and conventions, which override generic assumptions)

## 9. Response Format

- Lead with the answer, not preamble. No "Great question!", no "Certainly!".
- For code changes: show only the changed sections (with file path), not whole files, unless the file is small or new.
- After code: brief explanation of WHY the change works, not a line-by-line walkthrough.
- End with "Observations" section if you noticed issues outside the requested change (per section 4b).
- If you're uncertain about something, say so explicitly — do not present guesses as facts.
````

````markdown
# Git Commit Message Instructions

Generate commit messages following these rules.

## Format — Conventional Commits

```
<type>(<scope>): <subject>

<body>

<footer>
```

## Type (required)

- `feat` — new feature visible to users
- `fix` — bug fix
- `refactor` — code change that neither fixes a bug nor adds a feature
- `perf` — performance improvement
- `test` — adding/updating tests only
- `docs` — documentation only
- `style` — formatting, missing semicolons, etc. (no code change)
- `chore` — tooling, deps, build config
- `ci` — CI/CD config changes
- `revert` — revert a previous commit

## Scope (when applicable)

- The package or domain affected: `(accounts)`, `(transfers)`, `(design-system)`, `(auth)`, etc.
- For monorepo: prefer the package name from `package.json`.
- Omit scope only for repo-wide changes.

## Subject

- Imperative mood: "add", "fix", "remove" — not "added", "fixes".
- Lowercase first letter (unless it's a proper noun).
- No trailing period.
- ≤ 72 characters.

## Body (when the change is non-trivial)

- Explain **what** and **why**, not **how**.
- Wrap at 72 characters.
- Reference the ticket: `Refs: PROJ-1234` or `Fixes: PROJ-1234`.
- For banking: if the change has compliance/audit implications, mention them.

## Footer

- `BREAKING CHANGE: <description>` — for breaking API changes
- `Refs: <ticket-id>` — link to Jira/ADO/etc.
- `Co-authored-by: <name> <email>` — for pair work

## Examples

```
feat(transfers): add SEPA instant transfer option

Adds toggle in the transfer form to use SEPA Instant when
the recipient bank supports it. Falls back to standard SEPA
when not available.

Refs: BANK-4521
```

```
fix(accounts): correct balance rounding on multi-currency view

Balances were rounded before currency conversion, causing
1-cent discrepancies on USD→EUR display. Conversion now
happens before rounding.

Fixes: BANK-4490
```

```
refactor(design-system): extract Button focus styles to mixin

No visual change. Prepares for upcoming focus-visible spec
update in BANK-4612.

Refs: BANK-4600
```

## What NOT to do

- ❌ "fixed stuff", "updates", "wip", "minor changes" — useless
- ❌ Squash unrelated changes into one commit — split them
- ❌ Commit subject describing implementation: "use map instead of forEach" — describe outcome instead
- ❌ Reveal sensitive info in commit messages (account numbers, customer names, internal URLs)
````

````markdown
# React + TypeScript + Vite — Code Conventions

## Component Structure

- Functional components only. **Named exports** (no default exports except for Vite route/lazy modules).
- Props interface above component, named `<ComponentName>Props`. Export the props type if the component is exported.
- Order inside component body:
  1. Hooks (built-in first, then custom)
  2. Derived state / `useMemo` values
  3. Event handlers (`useCallback` only when passed to memoized children or to deps)
  4. Effects (`useEffect`, `useLayoutEffect`)
  5. Early returns (loading, error, empty)
  6. Render
- If a component crosses ~200 lines or 4 levels of JSX nesting, extract subcomponents or a custom hook.
- Custom hooks live in `hooks/` near the consumer, named `use<Name>`. Hooks > 30 lines of logic move to their own file.

## TypeScript

- `type` for unions, intersections, mapped/conditional types, function signatures.
- `interface` for object shapes meant to be extended or implemented.
- Discriminated unions for variant props (e.g. `{ variant: 'primary' } | { variant: 'destructive', confirmText: string }`).
- Do NOT use `React.FC` / `React.FunctionComponent` — type props directly.
- `as const` for readonly tuples, enum-like objects, and literal arrays used in type unions.
- No `enum` — use `as const` objects with a derived union type.
- Avoid `any`. If unavoidable, use `unknown` and narrow, with a comment explaining why.
- Avoid non-null assertions (`!`). Use type guards or early returns.

## State Management

- **Local state**: `useState` / `useReducer`. Use `useReducer` once you have 3+ related state values that change together.
- **Server state**: use the existing data-fetching library in this repo (likely React Query / TanStack Query or RTK Query — check imports before introducing anything else).
- **Derived state**: compute during render or in `useMemo`. Never mirror props/state into another `useState`.
- **Global state**: do NOT introduce a new global store. Use what already exists in the repo.

## Effects — `useEffect` discipline

- Effects are for synchronizing with external systems (DOM APIs, subscriptions, network). They are NOT for transforming data — that's derived state.
- Every effect must have a complete dependency array. Disabling `react-hooks/exhaustive-deps` requires a comment explaining the reason.
- Cleanup functions for: subscriptions, timers, event listeners, abortable fetches.
- If you find yourself writing `useEffect` to update state based on props, you probably want derived state instead — flag this.

## Data Fetching

- Always handle three states: loading, error, success. No assuming "it just works".
- Use `AbortController` / library-provided cancellation for in-flight requests on unmount or refetch.
- Type API responses at the boundary. Do not let `any` propagate from `fetch` into the rest of the app.

## Vite-specific

- Environment variables: `import.meta.env.VITE_*` only. Never `process.env`. Document new vars in `.env.example`.
- Lazy-load route components: `const Page = lazy(() => import('./Page'))` wrapped in `<Suspense>`.
- Asset imports: use `?url`, `?raw`, `?inline`, `?worker` query suffixes when needed.
- Path aliases: use the aliases configured in `vite.config.ts` / `tsconfig.json`. Do not introduce new ones without asking.

## Styling

- Use the design-system tokens and components first. Check `packages/design-system/` (or equivalent in this monorepo).
- Do not introduce inline styles for spacing, color, or typography that could come from tokens.
- Class names: follow the existing convention in the file (CSS Modules, styled-components, Tailwind, etc. — match what's there).

## Imports

- Order: (1) Node/external packages, (2) workspace packages (`@bank/*` or whatever the prefix is), (3) absolute aliases (`@/...`), (4) relative imports. Blank line between groups.
- No deep imports into other packages — use the package's public entry (`@workspace/pkg` not `@workspace/pkg/src/internal/foo`).
- Type-only imports use `import type` to keep the runtime bundle clean.

## Error Handling

- Wrap async UI operations in try/catch and surface user-facing errors via the existing error/toast mechanism (don't invent a new one).
- Error boundaries around routes and around any component that does data fetching. Match the existing `ErrorBoundary` API in the repo.
- Never swallow errors silently. If you intentionally ignore one, leave a comment with the reason.

## Forms

- Use the existing form library (React Hook Form, Formik, etc. — check before adding one).
- Validation schema co-located with the form, using whatever validator the repo uses (Zod, Yup, etc.).
- Disable submit during in-flight submission. Show loading state.
- All inputs must have associated labels (a11y).

## Accessibility (banking = audited for a11y)

- All interactive elements must be keyboard-reachable and have visible focus.
- Icons-only buttons need `aria-label`.
- Form inputs need labels (visible or `aria-label`/`aria-labelledby`).
- Color is never the sole means of conveying information.
- Modals: focus trap, return focus on close, `aria-modal="true"`, labelled by title.
- Use semantic HTML before reaching for ARIA. `<button>` not `<div onClick>`.
````

````markdown
# Code Craft — Universal Practices

These rules apply to ALL code you write or review.

## Comments

**Default: no comment.** Code must be self-explanatory through naming and structure. Reach for a comment only when it adds information the code itself cannot.

### ❌ Forbidden — comments that restate the code

```ts
// increment counter
counter++;

// loop through accounts
accounts.forEach(...)

// returns true if user is admin
function isAdmin(user) { ... }

// state for the modal
const [isOpen, setIsOpen] = useState(false);
```

If you find yourself writing a comment that paraphrases the next line, delete it and improve the name instead.

### ✅ Allowed — comments that add real information

- **Why, not what**: explain the reason for a non-obvious decision
  ```ts
  // Bank API returns dates as YYYYMMDD strings, not ISO — convert before use
  const date = parseLegacyBankDate(raw);
  ```
- **Workarounds**: link to a ticket / issue
  ```ts
  // Workaround for BANK-4521: focus is lost on Safari after dialog close
  setTimeout(() => triggerRef.current?.focus(), 0);
  ```
- **Spec/regulation references**: where a rule comes from external requirements
  ```ts
  // PSD2 SCA: re-authenticate if last auth > 5 minutes ago for transfers > €30
  ```
- **Type assertions and `any` justification** (required by `copilot-instructions.md`)
  ```ts
  // Library types are wrong; this is actually a string at runtime — see DefinitelyTyped issue #12345
  const value = response as unknown as string;
  ```
- **TODO / FIXME with ticket reference** — never anonymous TODOs
  ```ts
  // TODO(BANK-4612): replace with native :focus-visible once Safari 16 is min target
  ```

### JSDoc

- For exported functions/components in shared packages (especially design-system): JSDoc with description + params + return + example.
- For internal app code: JSDoc only when behavior isn't obvious from the signature. Don't JSDoc `function add(a: number, b: number)`.

## Naming

- **Full words over abbreviations**: `accountIndex` not `accIdx`, `customerProfile` not `custProf`. Industry-standard short forms are OK (`id`, `url`, `http`, `api`, `iban`, `bic`).
- **Booleans start with `is/has/should/can/will`**: `isLoading`, `hasError`, `shouldRefetch`, `canEdit`. Never just `loading` or `error` for booleans.
- **Functions are verbs**: `getAccountBalance`, `calculateInterest`, `formatCurrency`. Not `accountBalance` or `currency`.
- **React components are nouns in PascalCase**: `AccountSummary`, `TransferForm`.
- **Custom hooks start with `use` and describe the value/behavior**: `useAccountBalance`, `useDebouncedSearch`.
- **Avoid generic names**: `data`, `info`, `value`, `item`, `handle`, `process`, `manage`, `util` — every one of these means "I didn't know what to name it." Be specific: `transactions`, `customer`, `selectedAccount`.
- **Event handlers**: `onSubmit` for the prop, `handleSubmit` for the implementation in the component.

## Functions and Simplicity

- **Pure by default**: functions return values based on inputs, no side effects, unless side effects are the explicit purpose (event handlers, effects, API calls).
- **Single responsibility**: a function does one thing. If you write "and" in the name, split it.
- **Length**: aim for ≤ 30 lines. > 50 lines almost always means extract subfunctions.
- **Argument count**: ≤ 3 positional args. More → use an options object with named keys.
- **Early returns**: handle edge cases first, return early; main logic at indentation level 1.
  ```ts
  // ✅ Good
  function getDiscount(customer: Customer): number {
    if (!customer.isActive) return 0;
    if (customer.tier === 'standard') return 0;
    if (customer.tier === 'premium') return 0.1;
    return 0.2;
  }

  // ❌ Bad — pyramid of doom
  function getDiscount(customer: Customer): number {
    if (customer.isActive) {
      if (customer.tier === 'premium') {
        return 0.1;
      } else if (customer.tier === 'gold') {
        return 0.2;
      } else {
        return 0;
      }
    } else {
      return 0;
    }
  }
  ```

## DRY with Restraint — Rule of Three

- **Do NOT abstract on the first duplication.** Two similar pieces of code may diverge later; premature abstraction creates fragile shared code that resists divergence.
- **On the THIRD occurrence**, consider extracting. Even then, only extract if the duplications are *semantically* the same, not just structurally similar.
- Two functions that look the same but represent different domain concepts must stay separate. Banking example: `formatTransactionDate` and `formatStatementDate` may currently both do `format(date, 'yyyy-MM-dd')` — but they have different reasons to change. Don't merge.

## YAGNI — You Aren't Gonna Need It

Banking projects live 10+ years and accumulate dead code. Fight back:

- **No "just in case" parameters, options, or config flags.** Add them when a real second use case appears.
- **No abstract base classes, generic adapters, or plugin systems** without ≥ 2 concrete implementations that need them.
- **No premature generics.** `function getById<T>(id: string): T` is worse than `function getCustomerById(id: string): Customer` until you have 3+ entities.
- **No speculative interfaces.** If only one implementation exists, you don't need an interface yet.
- **Delete unused code aggressively.** Dead code is a maintenance burden and a comprehension burden.

## Magic Values

- **Magic numbers**: extract to named constants
  ```ts
  // ❌
  if (age > 18 && balance < 1000) { ... }

  // ✅
  const ADULT_AGE = 18;
  const MIN_INVESTMENT_BALANCE = 1000;
  if (age >= ADULT_AGE && balance < MIN_INVESTMENT_BALANCE) { ... }
  ```
- **Magic strings**: same. Especially for status enums, role names, currency codes.
- **Exception**: 0, 1, -1, 100 (percentages), 24/60/1000 (time) when usage is obvious from context.

## Performance

- **`useMemo` / `useCallback` are NOT defaults.** They have their own cost. Use them only when:
  - Profiling shows a real problem, OR
  - The value is passed to a memoized child (`React.memo`) and would break memoization, OR
  - The value is a dependency of another hook
- **`React.memo` is also not free.** Apply only to components that re-render often with same props (measured, not assumed).
- **Lists of > 100 items**: virtualize (react-window, react-virtual). Below that: don't.
- **Debounce/throttle**: search inputs, scroll handlers, resize handlers. Match the existing utility in the repo.
- **Bundle size**: avoid whole-library imports (`import _ from 'lodash'`). Use specific imports (`import debounce from 'lodash/debounce'`) or the modular package (`lodash-es`).

## Error Handling

- **Distinguishable errors**: subclass `Error` for domain errors so callers can `catch` specifically.
  ```ts
  class InsufficientFundsError extends Error { ... }
  class AccountFrozenError extends Error { ... }
  ```
- **Never throw strings or plain objects.** Always `throw new Error(...)` or a subclass.
- **Catch narrowly.** Don't `catch (e)` and assume it's a known type — narrow with `instanceof` or a type guard.
- **Don't swallow errors** unless intentional, and document the reason in a comment.
- **User-facing errors** must be safe to display. Never expose stack traces, internal IDs, or technical messages to end users (banking compliance).

## User-facing Strings

- Match the existing pattern in this repo for user-facing text. If strings live in a constants file or a centralized location, follow that. If they're inline, follow that — but flag heavy string duplication across components.
- **Validation messages, toast text, modal titles, form labels, button text, aria-labels** — keep them consistent in tone with the rest of the app. Don't introduce a new style.
- **Pluralization**: never `${count} items` style if the repo has a helper for it. If not, plain template literals are fine for now.

## Money — Special Care

- **NEVER use `number` for currency amounts.** Floating-point error compounds.
- Use the repo's money library (e.g. `dinero.js`, `big.js`, `bignumber.js`, or a custom `Money` type with bigint minor units — check `package.json`).
- All arithmetic on amounts goes through the library, not native `+ - * /`.
- Display always uses the repo's formatter, never string concatenation.

## Dates — Special Care

- **Never** ad-hoc date math: `new Date(date.getTime() + 86400000)` is wrong (DST, leap seconds). Use the repo's date library.
- **Always** be explicit about timezone. Banking transactions have legal timezone requirements.
- **Parsing**: never `new Date('2026-01-15')` — that's UTC. Use the library's parse function with format string.
- **Comparing**: dates can be `===` only by exact reference. Use library predicates (`isBefore`, `isSameDay`).

## Async and Race Conditions

- **AbortController** for cancellable requests. Always abort on unmount or when query changes.
- **Stale closures**: when an async callback reads state, that state may be stale. Use refs or pass state as args.
- **Loading state must be tied to the LATEST request**, not "any request". Track request IDs or use a query library that handles this.

## Security

- **Never `dangerouslySetInnerHTML`** unless the content is from a trusted, sanitized source (e.g. server-rendered markdown that's been DOMPurify'd).
- **Never trust user input.** Validate on the client AND server. Client-side validation is UX, not security.
- **CSP-friendly**: no inline event handlers in HTML strings, no `eval`, no `new Function()`.
- **Tokens**: never in `localStorage` / `sessionStorage` / cookies without `httpOnly` + `Secure` + `SameSite=Strict`.
- **PII in logs**: never. Even in dev. Use ID references, not values.
- **URL parameters**: never put PII (account numbers, names) in URLs — they get logged everywhere.

## Module Boundaries

- **Public API via `index.ts`** (barrel file). What's exported there is the contract.
- **Internal files are private** — other packages should not deep-import them.
- **Circular dependencies**: don't create them. If A imports B and B imports A, refactor (extract shared types to C).
- **Test files do not import from other test files.** Shared test utilities go in `test-utils/` or similar.

## Git Hygiene

- **One concern per commit.** Don't mix refactor + feature + dependency bump.
- **Refactors before features**, in separate commits, so reviewers can verify "no behavior change" first.
- **No commits with failing tests or type errors** — that's what `husky` is for.
- **No `WIP` commits on main branches.** WIP only on personal feature branches that get squashed.

## Definition of Done — Before opening a PR

A change is "done" when ALL of these are true:

- [ ] Tests written for new behavior (unit + E2E if user-visible flow)
- [ ] All tests pass locally (or, for E2E in this environment, you've maximally verified by inspection)
- [ ] TypeScript: zero errors, zero new warnings
- [ ] ESLint: zero errors
- [ ] No `console.log` / `debugger` left
- [ ] No commented-out code (delete it; git remembers)
- [ ] A11y: keyboard navigable, labels present, focus management correct
- [ ] No new dependencies without team approval
- [ ] Commit messages follow Conventional Commits
- [ ] Self-reviewed: read your own diff before requesting review

If you're suggesting code, mention which of these are "done" vs "still needed".
````

````markdown
# Unit Tests — Vitest + React Testing Library

## Philosophy

Test **behavior visible to the user**, not implementation details. A test should fail when the user-facing behavior breaks, and pass when the implementation is refactored without changing behavior.

## Forbidden in unit tests

- Testing component state directly (no `wrapper.state()`, no asserting on `useState` values via spies)
- Spying on internal hook calls just to check they ran
- Testing CSS class names as a proxy for behavior (test the visible effect instead)
- `setTimeout`, `setInterval`, `wait(N)` with arbitrary durations — use `vi.useFakeTimers()` or `waitFor`
- `fireEvent` for user interactions — use `userEvent` (it dispatches the full event sequence: pointerdown, mousedown, focus, click, etc.)
- Snapshot tests for feature components. Snapshots are OK only for stable design-system primitives where the entire rendered output IS the contract.
- Mocking modules you own. If you find yourself doing this, the seam is wrong — refactor to inject the dependency.

## Required patterns

- **Selectors priority**: `getByRole` (with accessible name) > `getByLabelText` > `getByText` > `getByTestId`. Reach for `getByTestId` only when nothing else identifies the element accessibly — and if you need it, add `data-testid` to the component first.
- **One concept per test**, but multiple `expect()` calls allowed if they verify the same concept.
- **AAA structure**: arrange, act, assert. Separate with blank lines, not comments.
- **HTTP mocking**: use MSW (Mock Service Worker) at the network boundary, not module mocks of axios/fetch. Match the existing MSW setup in this repo.
- **`userEvent` setup**: `const user = userEvent.setup()` inside each test (or `beforeEach`), not at module level.
- **Async assertions**: `findBy*` queries (which `waitFor` internally) for elements that appear after async work. `waitFor(() => expect(...))` for non-element assertions.

## Test descriptions

- `describe('ComponentName', ...)` for the unit under test.
- `it('should <behavior> when <condition>')` — describe behavior, not implementation.
  - ✅ `it('shows an error message when submitting an empty form')`
  - ❌ `it('calls setState with error')`
  - ❌ `it('renders correctly')` (meaningless)

## Coverage rules

- Every exported function/component must have at least: happy path, one error/edge case.
- Conditional rendering: test each branch.
- User interactions that change UI: test the visible result, not the handler call.
- Custom hooks: test via a tiny test component or `renderHook` from `@testing-library/react`.

## Mocking discipline

- Mock at module boundaries you don't own (HTTP, browser APIs, third-party SDKs).
- For dates: `vi.useFakeTimers({ now: new Date('2026-01-01') })` — reset in `afterEach`.
- For randomness: inject a seed/generator, don't `vi.spyOn(Math, 'random')`.
- Reset mocks between tests: `vi.resetAllMocks()` in `beforeEach` (or configure globally in `vitest.config.ts`).

## Before returning a test

Confirm in your reply:
- Test name describes user-visible behavior, not implementation
- Selectors prefer roles/labels over test IDs
- No arbitrary timeouts
- Async work uses `findBy*` or `waitFor`
- No assertions on internal state
- Cleanup (timers, mocks) is handled
````

````markdown
# E2E Tests — Playwright (HIGH STAKES)

## CRITICAL Environment Constraint

I write E2E tests in **VS Code in the browser (cloud workspace)**. Chromium is **NOT installed locally**. I cannot run `npx playwright test` to verify them. They run only in CI.

**This means**: every test you write must be correct on the first attempt. You cannot rely on me catching errors locally. Treat every E2E test as if it goes straight to a PR.

## Defense-in-depth checklist — apply to EVERY E2E test

### Before writing the test, you MUST have:

- [ ] The actual source of the component(s) under test (or you must ask me for it)
- [ ] An example existing E2E test from the same repo (for fixtures, auth, page objects)
- [ ] The actual route(s) the test will navigate to
- [ ] The exact text/role/label of every element you'll select

If you don't have these, ASK before writing the test. Do NOT guess selectors.

### When writing the test:

- [ ] Every selector verified against the component source — quote the line of source code that confirms each selector
- [ ] No `page.waitForTimeout(N)`. Use `expect(locator).toBeVisible()`, `expect.poll()`, or `waitFor` with conditions.
- [ ] No CSS selectors with class names. Class names are not stable contracts.
- [ ] No xpath.
- [ ] No `nth-child` or positional selectors that break when the page changes.
- [ ] No selectors that match dynamically-rendered text (server-provided strings, formatted numbers/dates) — text content can change without code changes.

## Selector Strategy (in priority order)

1. **`page.getByRole('<role>', { name: '<accessible name>' })`** — preferred. Stable, semantic, a11y-aligned.
2. **`page.getByLabel('<label text>')`** — for form inputs.
3. **`page.getByTestId('<id>')`** — only when role/label can't uniquely identify. If the component lacks a `data-testid`, ASK me to add one rather than reaching for fragile alternatives.
4. **`page.getByText('<text>', { exact: true })`** — only for stable, hardcoded UI text (brand names, fixed labels). Never for text that comes from a server response or could change without a deliberate code change.

## Required Test Structure

- **One `test.describe` per user flow** (e.g. "Account opening", "Wire transfer").
- **`test.beforeEach`** for auth and shared setup. Reuse fixtures from `e2e/fixtures/` — do not invent your own auth flow.
- **Page Object Model**: check `e2e/pages/` (or equivalent) for existing POs. Reuse them. If a needed PO doesn't exist, ask before creating one — I want to discuss the API.
- **Network mocking**: `page.route()` matching the patterns already used. Most banking flows have predefined fixtures for account data, transactions, etc. — find and use them.
- **Test data**: factories from `e2e/factories/` (or equivalent). Never hardcode account numbers, IBANs, customer IDs.

## Forbidden in E2E

- `page.waitForTimeout()` — always
- `await page.keyboard.press('Tab')` followed by an assumption about focus position — use focus selectors
- Tests dependent on test execution order (each test must be independent)
- Asserting on toast/notification text by exact match when the text comes from a server response or formatter — match by role (`getByRole('alert')`) and verify the kind of message instead
- `force: true` on clicks unless you can explain why the element is intentionally non-interactive
- Hardcoded waits between actions
- `expect(locator).toHaveText(...)` against text that comes from server data or formatters (use role/structure assertions or compare to the same source the UI used)

## Required in E2E

- `await expect(locator).toBeVisible()` before interacting — Playwright auto-waits, but explicit asserts make failures readable
- Handle navigations: `await page.waitForURL(/.../)` after actions that route
- Network mocks set up BEFORE navigation that triggers the request
- Cleanup: any data created during the test must be cleaned up in `afterEach` or via test isolation

## Banking-specific E2E concerns

- **Multi-factor auth flows**: use the existing test bypass / fixture, never automate real OTP/SMS.
- **Money amounts**: use the formatting helpers used in production code; formatting affects decimal/thousands separators, so don't hardcode amount strings in assertions.
- **Sensitive data masking**: tests should verify masking, not the unmasked value (e.g. `**** 1234` for card numbers).
- **Session timeout**: long flows may hit session expiry — confirm the fixture handles this.

## Before returning E2E code — REQUIRED CONFIRMATION

End your response with this checklist, filled in:

```

- Source of truth for selectors: [filename(s) you read]
- Selectors verified against source: [yes/no — if no, do NOT proceed]
- Existing fixtures/POs reused: [list them]
- No timing-based waits: [yes]
- Will run in headless CI Chromium: [yes — explain any browser-specific risks]
```

If you cannot fill in any line truthfully, STOP and ask me for the missing information instead.
````
